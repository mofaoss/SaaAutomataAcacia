# SaaAssistantAca 架构（框架驱动）

## 1. 核心理念：Framework vs. Features

该项目已从传统的分层式应用转型为 **Framework-Feature** 模型。

* **Framework（`app/framework`）**：提供“游戏规则”（渲染、执行、可观测性、i18n），本身不绑定任何具体功能。
* **Features（`app/features`）**：承载垂直业务逻辑（模块）。各功能模块通过装饰器和标准化元数据“挂接”到框架中。

---

## 2. 结构分层

### 📂 `app/framework/`（内核）

* **`core/`**：系统的大脑。

  * `module_system/`：负责模块发现、`@periodic_module` / `@on_demand_module` 装饰器，以及 `config_schema` 生成。
  * `task_engine/`：管理线程生命周期、调度器，以及 `BaseTask` 的执行。
  * `event_bus/`：通过发布/订阅机制解耦组件（例如 UI 日志、任务状态更新）。
* **`application/`**：业务层编排。

  * `modules/`：统一 UI 消费模块规格（`ModuleSpec`）的方式。
  * `periodic/` ：定义“任务”的状态机。如：日常任务，即时任务，计划周期等。
  * `startup/` ：界面加载方案。
* **`ui/`**：渲染引擎。

  * `auto_page/`：通过 `AutoPageFactory` 基于模块元数据动态生成 Qt 页面。
  * `widgets/` 与 `themes/`：共享 UI 组件与视觉样式。
* **`infra/`**：技术适配层。

  * `automation/`：窗口控制、输入模拟。
  * `vision/`：OCR 与图像识别服务。
* **`i18n/`**：高级国际化系统，支持动态模板和模块级作用域。

### 📂 `app/features/`（业务）

* **`modules/`**：按功能垂直切分的模块（例如 `fishing`、`chasm`）。

  * 结构：每个模块都是一个包含业务逻辑（usecases）的独立包。
  * 接入方式：模块通过 Python Type Hints 和 `Field()` 元数据定义其配置。

---

## 3. 关键机制

### ⚙️ AutoPage UI 生成（“零 UI 代码”流程）

1. **声明**：某个功能模块使用 `@periodic_module`，并定义变量（例如 `count: int = Field(default=5)`）。
2. **元数据提取**：框架中的 `ModuleMeta` 提取这些类型和约束信息。
3. **渲染**：`AutoPageFactory` 实例化页面，`AutoPageBase` 遍历 `config_schema` 并完成映射：

   * `bool` -> `QCheckBox`
   * `int` -> `QSpinBox`
   * `str` -> `QLineEdit` / `QComboBox`
4. **绑定**：`AutoPageActionsMixin` 会自动将 UI 中的“运行”按钮绑定到模块的 `runner` 方法。

### 🧵 任务执行模型

* **线程隔离**：所有自动化逻辑都运行在独立的 `TaskThread` 中，以防止 UI 卡死。
* **状态跟踪**：`RuntimeSession` 维护当前执行过程的全局状态。
* **通信方式**：模块通过 `EventBus` 信号与 UI 通信，而不是直接持有 UI 引用。

---

## 4. 国际化（i18n）

系统采用独特的 **运行时翻译** 流程：

* **Keys**：开发者在装饰器中使用英文 key / label。
* **Resolution**：`AutoPageI18nMixin` 通过 `i18n/runtime.py` 在运行时动态翻译这些 key。
* **Injection**：支持使用 `TemplateRenderer` 在翻译中注入动态变量。

---

## 5. 目录清理（移除遗留结构）

以下遗留路径已被 **完全移除**，不得继续使用：

* `app/presentation`（已由 `app/framework/ui` 替代）
* `app/view` 与 `app/repackage`（已废弃）
* 根目录下的 `app/core`（已迁移至 `app/framework/core`）
* `app/modules`（已迁移至 `app/features/modules`）

---

## 6. 开发规则

* **不要直接编写 UI**：除非需求非常复杂难以满足，应使用 `AutoPage` 系统。
* **类型安全**：始终为配置变量提供明确的类型提示和 `Field` 描述。
* **解耦**：功能模块绝不能从 `app/framework/ui` 导入内容。模块只提供数据与逻辑，显示由框架负责。
* **运行时路径规范**：始终使用标准化的 `runtime/appdata`、`runtime/log` 和 `runtime/temp` 目录。
